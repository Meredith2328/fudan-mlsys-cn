# 第 7 章 深度学习编译器：从通用性到专用化，再回到学习式搜索

## 1. 为什么深度学习一开始就需要编译器

深度学习模型写在“语义”的语言里。我们谈注意力、前馈网络、层归一化、卷积、激活函数和损失函数；但硬件并不执行语义，硬件执行的是指令。它负责在不同存储层之间搬运字节，调度 warp，发出 Tensor Core 指令，解决依赖关系，并在数据不能按时到达时发生停顿。

深度学习编译器存在的原因，正是这两个世界之间的距离太大。

在模型层，我们描述的是“想算什么”；在硬件层，我们必须决定“怎样把这件事高效地落实到某台具体机器上”。这两层之间并不是一个小小的语法落差，而是一个规模很大、结构很强的优化问题。编译器就是负责填平这道鸿沟的系统。

对深度学习来说，这个问题之所以在近十年变得重要，是因为神经网络终于大到足以有用，也大到足以让执行效率在生产环境中变成核心问题。模型从研究原型走向真实系统之后，问题就不再只是“能不能跑起来”，而是“能否跑得足够快、是否装得进内存、能否把硬件吃满、能否适应真实部署环境的规模与约束”。

因此，本章的核心判断可以压缩成一句话：

> 深度学习编译器，是一个把高层张量计算映射到硬件执行过程中的大规模结构化优化问题自动化的软件系统。

而且，这种映射从来不是静态不变的。工作负载在变，硬件在变，瓶颈也在变，于是主导性的编译器设计也随之变化。过去十年里，深度学习编译器大体可以沿着四个里程碑来理解：

1. **LLM 之前的编译器时代**，最典型的代表是 **TVM**。那时的核心问题是“很多模型 × 很多硬件后端”。
2. **LLM 时代早期的收敛**，最典型的代表是 **Triton**。问题开始坍缩为“Transformer 工作负载在 NVIDIA GPU 上的高效实现”。
3. **显式控制的回归**，最典型的代表是 **TileLang**。随着内核越来越专门化，仅靠黑盒自动化已不够用。
4. **学习式搜索的新阶段**，这里的代表是 **CUDA-L2**。大语言模型加强化学习开始重新接管内核优化，但方式与过去已经不同。

这些系统之所以值得放在同一章里讨论，不是因为它们只是“不同工具”，而是因为它们分别回答了同一个更深的问题：

> 给定一个张量工作负载和一台目标机器，到底应该暴露多少结构、隐藏多少结构，以及谁来搜索优化空间：程序员、编译器，还是学习型智能体？

## 2. 在编译器出现之前，真正被优化的到底是什么

在讨论编译器之前，首先要说清楚：我们到底在优化什么。

当我们前面学习 GPU 编程时，其实已经看到过这个问题的原型。一个张量计算可以被看成一个**迭代空间（iteration space）**，也就是所有循环索引组合形成的工作集合。以矩阵乘法为例，

\$$C[i,j] = \sum_k A[i,k]B[k,j],$$

一种自然实现就是：

```text
for (int i = 0; i < M; ++i) {
  for (int j = 0; j < N; ++j) {
    float acc = 0.0f;
    for (int k = 0; k < K; ++k) {
      acc += A[i][k] * B[k][j];
    }
    C[i][j] = acc;
  }
}
```

在这个意义上，GPU 内核本质上就是：把迭代空间映射到硬件执行层次与存储层次上的一种实现方式。

这里有两个始终占主导地位的目标。

### 2.1 并行性

第一个目标是把一个大的计算拆成许多较小、但算术强度足够高的子任务，使它们能够在大量处理单元上并发运行。在 GPU 上，这些处理单元以层次化方式组织：grid、block、warp、thread。并行化做得好，算术单元就会持续忙碌；并行化做得差，算术资源就会空转。

### 2.2 局部性

第二个目标是减少数据移动成本。算术和存储在物理上是分离的：寄存器很快但很小，共享内存和 L1 也快但容量有限，HBM 容量大却远得多、慢得多。一个高性能内核的关键，是让可重用的数据尽量长时间停留在层次结构的上层。

这也解释了为什么看起来五花八门的优化技巧，最终几乎都能归结为同一句话：**让每搬运一个字节都做更多有用计算。**

### 2.3 典型的优化旋钮

课程里提到的优化手段，并不是一堆互不相关的小技巧，而是深度学习编译器一直围绕着展开的标准自由度：

- **平铺（tiling）**：把迭代空间切成更小的块。以 GEMM 为例，可以把输出空间切成例如 \(128 \times 128\) 的 tile，同时对归约维度分块。平铺既能提高局部性，也能提高并行度。
- **内存放置（memory placement）**：决定中间数据放在哪一级存储里，是全局内存、共享内存、寄存器，还是某种更专门的片上存储。
- **循环重排（loop permutation）**：数学计算不变，但重用模式会大幅变化。
- **融合与切分（fusion / fission）**：融合通常减少中间结果回写 HBM 的成本；切分有时则能改善并行性、降低寄存器压力或简化调度。
- **循环展开（unrolling）**：减少循环控制开销，提高指令级并行性，但会增加代码体积和寄存器压力。
- **线程映射（thread mapping）**：决定哪一部分迭代空间交给哪一个 block、warp 或 thread 计算，这会直接影响访存合并、数据重用、occupancy 和同步模式。
- **同步（synchronization）**：共享数据时必须同步才能保证正确性，但同步本身也有开销。

所以，在编译器真正接管优化之前，问题本身就已经是一个高维、耦合、离散的设计空间。

## 3. GPU 内核优化本质上是一个结构化离散搜索空间

课程里最重要的抽象之一，是把这些选择统一理解为一个**结构化的离散搜索空间**。

一个内核的优化从来不是靠单个参数决定的，而是依赖一整组决策的组合：tile 大小、循环顺序、展开因子、线程绑定方式、缓存策略、同步方式、是否 tensorize、如何布局共享内存，等等。很多变量本身就是离散的：tile 大小只能从有限集合里选，循环顺序本身是排列问题，展开因子是整数，线程映射和同步方式则更像类别选择。

更难的是，这些变量彼此强耦合：

- tile 变大可能提升重用，但也可能把共享内存撑爆；
- 展开更多可能提升指令级并行性，但同时抬高寄存器压力；
- 某个循环顺序可能改善局部性，却让向量化更难；
- 更激进的融合可能减少 HBM 流量，却压低 occupancy；
- 更好的线程映射可能改善合并访问，却又在别处引入 warp 发散。

也就是说，性能不是一系列独立决策的简单叠加，而是一个充满交互项的曲面。

这就是为什么这个问题常被描述为“组合爆炸”的原因。实践中，编译器只能依赖启发式规则、剪枝、整数规划、混合整数规划、代价模型、自动调优或学习式搜索等手段来逼近最优解。

也正因为整个问题太大，才需要编译器栈来做分层分解。

## 4. 为什么必须有编译器栈

深度学习编译器之所以是“栈”，是因为同一个计算在不同层面需要不同的表示。

在最上层，计算对人是有意义的；在最下层，计算对硬件才是有意义的；中间每一层保留的结构都不同。某些优化需要保留全局依赖信息，另一些优化则需要显式看到循环、缓存、线程与指令级细节。

一个典型的深度学习编译器栈大致包含：

1. **模型层（model layer）**：表达神经网络的功能结构。
2. **图 IR（graph IR）**：表达算子依赖与数据流关系。
3. **张量 IR（tensor IR）**：表达循环、缓冲区、访存和 schedule 空间。
4. **硬件 IR（hardware IR）**：表达面向具体设备的执行形式。
5. **运行时（runtime）**：处理实际的动态启动、内存管理和协调行为。

这不是官僚式堆叠，而是必要分解。

模型层告诉我们“算什么”；图 IR 告诉我们“哪些算子依赖哪些算子”；张量 IR 告诉我们“循环如何组织、内存如何访问”；硬件 IR 告诉我们“这些工作怎样映射到真实机器”；运行时则决定“这个编译好的计划如何在当前环境里被启动和协调”。

性能来自于：在每一层都保留恰当的结构，并在恰当的层上做恰当的优化。

## 5. 历史转变：LLM 之前与 LLM 之后

理解深度学习编译器的历史，最清晰的方式是把它分成两个大阶段。

### 5.1 LLM 之前：模型多样、硬件多样

大约在 2017 到 2020 年间，深度学习仍然高度多样化。模型既有 CNN、RNN、LSTM、GAN，也有早期的注意力模型与各种混合结构；硬件后端同样高度分裂：NVIDIA GPU、Google TPU、ARM CPU、FPGA、移动端加速器、Ascend、IPU 等并存。

因此，那一阶段编译器面对的核心问题是：

> 如何让一个模型描述能够在尽可能多的硬件上高效运行？

这正是 TVM 最合理的历史语境：问题的关键词是**通用性**。

### 5.2 LLM 之后：工作负载收敛，压力转向极致效率

2020 年之后，局面发生了明显变化。工作负载开始向 Transformer 体系高度收敛，硬件也越来越围绕 NVIDIA GPU 与 CUDA 生态集中。这在某种意义上简化了编译器问题：我们不再需要“为所有模型和所有硬件同时优化一切”。

但这种收敛并没有让问题变简单，它只是把压力集中到另一个方向：**每一个百分点的执行效率都意味着巨大的基础设施成本差异。**

在这个阶段，Triton 之所以出现，是因为模型开发者迫切需要一种能够在 Python 中写出接近 CUDA 性能的自定义 GPU 内核、但又不用彻底变成 CUDA 专家的方式。

再往后，问题又进一步变化。随着 FlashAttention、GQA、MLA、量化内核、复杂融合 epilogue 等特化内核不断出现，单纯依赖一个“漂亮的高层 DSL”已经不够。于是 TileLang 重新把更多控制权交还给用户。

最后，当显式暴露出来的优化空间重新变得太大、太细、太依赖硬件细节之后，学习式搜索开始登场。CUDA-L2 的意义就在这里：不是继续把更多旋钮交给人，而是让机器重新接管这些旋钮。

## 6. TVM：LLM 之前最典型的深度学习编译器

### 6.1 TVM 试图解决的历史问题

TVM 可以说是 pre-LLM 时代最典型的深度学习编译器。它诞生时的痛点非常具体：框架如 TensorFlow、Caffe、早期 PyTorch，高度依赖为特定硬件手工编写的优化内核。这样一来，“很多算子 × 很多后端”的支持矩阵会迅速爆炸，维护成本极高，扩展也很慢。

TVM 的回答是：构建一个统一的编译器栈，从高层模型出发，逐层下降到具体后端，并尽可能自动生成高效代码。它试图把“深度学习优化”从手工维护的大量专用内核，转变成一个可编程、可搜索、可自动化的编译过程。

### 6.2 TVM 的栈式组织：图、张量、代码生成、运行时

TVM 的重要性，不只在于某个优化技巧，而在于它把编译问题明确拆成多个层面。

- **图层**：关注整个模型结构，处理算子连接关系、融合、常量折叠、内存规划等。
- **张量程序层**：关注单个算子如何以 loop nest、buffer、memory scope 的形式实现，进而进行 tiling、reorder、parallelize、vectorize、tensorize、cache 等调度。
- **后端层**：把前面的结果进一步 lower 到 LLVM IR、CUDA、ROCm 等设备相关代码。
- **运行时层**：负责模块加载、内存管理、启动与协调。

历史上，TVM 的图级抽象常由 Relay 承担；后来逐渐向 Relax 过渡，以支持动态图和更灵活的模型表达。张量侧也从早期的 Tensor Expression（TE）逐步演进到更显式、更便于变换的 TensorIR（TIR）。

### 6.3 compute-schedule 分离：TVM 的核心思想

TVM 最重要的思想之一，是**compute-schedule 分离**。

compute 负责描述“这个算子在数学上算什么”，schedule 负责描述“如何在机器上执行它”。

例如，矩阵乘法的 TE 风格定义，可以概念化地写成：

```text
k = te.reduce_axis((0, K), name="k")
C = te.compute(
    (M, N),
    lambda i, j: te.sum(A[i, k] * B[k, j], axis=k)
)
```

这里表达的是“要算什么”，并没有决定 tile 大小、循环顺序、线程组织或缓存方式。

而 schedule 则进一步说明“怎么做”：

```text
s = te.create_schedule(C.op)
i, j = s[C].op.axis
io, ii = s[C].split(i, factor=32)
jo, ji = s[C].split(j, factor=32)
s[C].reorder(io, jo, ii, ji)
s[C].parallel(io)
s[C].vectorize(ji)
```

在这个小例子里，已经能看到深度学习编译器最核心的几种思想：

- `split` 对应 tiling；
- `reorder` 改变循环顺序，从而改变局部性与向量化机会；
- `parallel` 暴露并行工作；
- `vectorize` 把内层连续访问映射到 SIMD 或更宽访存。

TensorIR 进一步把这种思想系统化：程序显式包含 buffer、loop 和 block；schedule 则变成对这些结构做变换的过程。这样，正确性和性能就被清楚地拆开了：算子语义保持不变，但执行方式可以被广泛搜索。

### 6.4 图级优化：为什么不仅仅是“生成 kernel”

TVM 并不把优化局限在单个算子内部。图级优化同样重要，因为很多性能问题来自算子之间的边界。

例如，如果 `MatMul -> Add -> GELU` 作为三个独立算子执行，就意味着中间结果会被写出到全局内存，再重新读回；而如果它们能在图级上被合法融合，那么中间值就可能停留在寄存器或片上缓存里，从而大幅减少 HBM 流量和 kernel launch 开销。

课程里用 `Conv2D -> BiasAdd -> ReLU -> BatchNorm` 这样的例子说明这一点也非常典型：卷积是算术密集的，而后面的若干 elementwise 或近似 elementwise 操作更偏带宽受限。如果把它们拆开跑，往往是在浪费存储层次；把它们适度融合，则可能显著降低内存流量。

### 6.5 张量层优化：这其实就是结构化的 CUDA 优化空间

一旦进入 TensorIR，程序就开始显式包含 loop、buffer 和 memory scope，这时 TVM 的优化空间就与我们在 CUDA 中见到的内核优化问题高度相似了。

张量层常见的 schedule primitive 包括：

- loop splitting / tiling
- loop reordering
- parallelization / binding
- vectorization
- caching（如 `cache_read`、`cache_write`）
- tensorization
- software pipelining

从某种意义上说，TensorIR 就是在用一种更系统、更可变换、更适合编译器处理的形式，把“CUDA 优化空间”编码出来。

### 6.6 AutoTVM 与 MetaSchedule：为什么必须自动搜索

手工写 schedule 很快就会遇到可扩展性问题，因此 TVM 很早就把自动调优纳入核心。

AutoTVM 的基本思路是：

1. 用模板定义 schedule 空间；
2. 生成候选 schedule；
3. 在真实硬件上编译并运行；
4. 测量延迟；
5. 训练或更新代价模型；
6. 再用代价模型引导下一轮搜索。

这就是典型的 autotuning 闭环。

MetaSchedule 则把这个思路进一步向 TensorIR 原语空间推进。它不再过度依赖人类手写的 schedule 模板，而是直接在 IR 变换空间里搜索。这一变化的意义非常大：当新的调度原语（例如更复杂的 tensorization、loop partitioning、pipeline 变换等）出现时，搜索系统不必为每个算子重写模板，就能吸纳这些新自由度。

### 6.7 TVM 真正重要在哪里

TVM 真正改变的，是我们看待机器学习系统的方式。它让人们第一次大规模地把“优化 kernel”理解成一个可编程、可组合、可部分学习的编译过程，而不只是某些专家手工写出来的黑魔法代码。

它之所以适合 pre-LLM 时代，正是因为那个时代需要的是：

- 跨模型的图级优化；
- 跨算子的张量级调度；
- 跨硬件后端的代码生成；
- 靠自动调优来勉强应对支持矩阵的爆炸。

当然，它的通用性也带来了代价：栈更深、跨层相互作用更复杂、调试更困难。但这并不是 TVM 的偶然缺点，而是“广义编译器问题”本身的代价。

## 7. Triton：LLM 时代早期最关键的编译器

### 7.1 Triton 为什么会出现

Triton 属于一个与 TVM 明显不同的时代。

到了 LLM 时代早期，问题已经大大收敛：主要工作负载是 Transformer，主要硬件平台是 NVIDIA GPU。于是，“支持一切模型、支持一切后端”不再是最优先的问题。最优先的问题变成了：如何让主要在 Python 中工作的模型开发者，能够写出高性能自定义 GPU 内核，而不必成为专职 CUDA 专家。

因此，Triton 的目标并不是“优化一切、部署到一切地方”，而更接近于：

> 给 Python 用户一门高生产力的 GPU kernel 语言，让他们在屏蔽大部分底层复杂性的同时，获得接近 CUDA 的性能。

### 7.2 Triton 的程序模型：以 block 程序为核心

Triton 的核心抽象是：程序员按**block 级程序**来写 kernel。

一个 Triton kernel 通常由 `@triton.jit` 装饰；程序员写出的逻辑，是“一个程序实例负责一个数据块”的逻辑。底层的 block 到 CUDA 执行层次的映射，则由 Triton 编译器和运行时去处理。

这与 CUDA 的关键差别在于：

- CUDA 显式暴露线程、块、共享内存、寄存器等概念；
- Triton 更多地把“块程序（program）”作为主要用户可见抽象；
- CUDA 常要求程序员手工处理大量低层细节；
- Triton 则自动处理更多线程级与内存级细节。

### 7.3 `@triton.jit`、`program_id`、`arange` 与 `mask`

即使在一个最简单的向量加法例子里，也能看出 Triton 的核心构件：

- `@triton.jit`：标记函数进行 JIT 编译；
- `tl.program_id(axis)`：取得某个维度上的程序实例编号；
- `tl.arange`：生成块内偏移；
- `tl.load(..., mask=...)` / `tl.store(..., mask=...)`：实现带边界保护的向量化访存。

概念性代码大致是：

```text
@triton.jit
def kernel(x_ptr, y_ptr, z_ptr, N, BLOCK: tl.constexpr):
    pid = tl.program_id(0)
    offsets = pid * BLOCK + tl.arange(0, BLOCK)
    mask = offsets < N
    x = tl.load(x_ptr + offsets, mask=mask)
    y = tl.load(y_ptr + offsets, mask=mask)
    z = x + y
    tl.store(z_ptr + offsets, z, mask=mask)
```

应当这样理解它：

- 一个 program instance 负责一个长度为 `BLOCK` 的数据块；
- `tl.arange` 给出块内 lane 偏移；
- `mask` 解决边界问题；
- 程序员主要写的是向量式 block 逻辑，而不是显式 thread 逻辑。

这就是 Triton 最关键的抽象跃迁：它把 thread 级并行隐藏进了更高层的 block 级计算里。

### 7.4 Triton 为什么特别适合 GEMM、Softmax 和 Attention

Triton 的教程和成功案例几乎都集中在 GEMM、fused softmax、fused attention、grouped GEMM、persistent matmul 等内核上，这并不是偶然。

这些算子恰好有几个共同特征：

- 结构规整；
- block 级表达自然；
- 与 NVIDIA GPU 的执行模型高度匹配；
- 通过 fusion 能显著减少 HBM 流量；
- 在较少暴露底层细节的情况下，仍能获得非常强的性能。

因此，在 LLM 早期这个“工作负载收敛、硬件也收敛”的窗口期，Triton 的确能在开发效率与性能之间给出非常漂亮的平衡。

### 7.5 Triton 的局限

Triton 的强项正来自它对优化空间的“收缩”。但一旦 kernel 开始变得更不规则、更加特化、更加依赖低层布局与硬件特性，这种收缩就会变成限制。

程序员很难在 Triton 中直接精细控制：

- 共享内存的精确布局；
- swizzle 模式；
- warp specialization；
- 更复杂的软件流水线；
- 某些非常规的融合与 staging 结构。

换句话说，Triton 的优雅来源于它“隐藏了足够多”；而它的边界，也恰恰出现在“被隐藏的部分开始重新重要起来”的地方。

## 8. TileLang：当控制旋钮重新回到程序员手里

### 8.1 TileLang 为什么会出现

TileLang 正是在 Triton 的抽象开始不够用的时候出现的。

随着 LLM 内核越来越专门化、GPU 本身也越来越复杂，很多场景下程序员真正需要的已不再是“写一个很快的通用自定义 kernel”，而是“写一个对布局、流水线、线程绑定、Tensor Core 映射和 shape 依赖行为都高度敏感的专门 kernel”。

TileLang 的回应并不是抛弃抽象，而是把抽象向下移动一层：

> 仍然以 tile 为思考单位，但把围绕 tile 的调度、布局和内存控制更多地显式暴露出来。

### 8.2 tile 被提升为第一类对象

TileLang 的核心是让 tile 成为真正的一等公民。一个 tile 是一块有形状的数据，可以被 warp、线程块或其他并行单元拥有和操纵。

这与 Triton 的差别很关键：在 Triton 中，程序员写 block 级逻辑，但很多围绕 block 的调度细节仍然被隐藏；在 TileLang 中，程序员不只是知道“有 tile”，而是可以更直接地控制 tile 如何放进共享内存、如何进 fragment、如何与 pipeline 结合、如何绑定到线程层次。

### 8.3 TileLang 显式暴露了什么

TileLang 构建在 TVM 风格基础设施之上，但比 Triton 更积极地把调度空间暴露给用户。

典型的用户可见原语包括：

- `T.Kernel(...)`：定义 kernel 启动上下文；
- `T.alloc_shared(...)`：分配共享内存；
- `T.alloc_fragment(...)`：分配寄存器 fragment；
- `T.copy(...)`、`T.async_copy(...)`：显式表达跨存储层的数据搬运；
- `T.Pipelined(...)`：表达软件流水线；
- `T.Parallel(...)`：表达并行拷贝或并行循环；
- `T.gemm(...)` 等计算原语；
- layout / swizzle 等布局控制工具。

这些能力意味着：TileLang 不只是“能写算子”，而是允许程序员在 tile 周围明确构造一整套调度骨架。

### 8.4 三层控制：从舒适区到专家区

TileLang 的设计一个很重要的点，是它不只提供一层抽象。它往往允许程序员从相对高层的 tile 逻辑逐步下降到更低层、更加接近手写 CUDA/HIP 的控制模式。

这件事很重要，因为它表明 TileLang 不是简单的“Triton 加更多 knob”，而是在尝试覆盖一个更宽的“生产力—控制力”区间：既能给普通用户较高层抽象，也能让专家为极端内核深入到底层细节。

### 8.5 内存布局为什么是 TileLang 的关键例子

课程特别强调了一个点：TileLang 最重要的价值之一，是让用户重新获得对**内存布局**的精细控制。

在困难 kernel 里，shared memory 中 tile 的排布、寄存器 fragment 的布局、padding 与 swizzle 的选择、Tensor Core 期望的片段组织方式，都会显著影响 bank conflict、coalescing、alignment 和 tensorization 效果。

这也是为什么很多高性能 kernel 一旦超出 Triton 的舒适区，就会迫使人重新思考更显式的布局控制。

### 8.6 软件流水线与 warp specialization 的回归

TileLang 另一个重要暴露点，是 pipeline 控制。

很多真正高性能的 kernel 都依赖“搬运下一块数据”与“计算当前块数据”的重叠，也就是显式的软件流水线。Triton 往往通过 `num_stages` 这类参数间接暴露这件事，而 TileLang 则让用户更直接地表达 pipeline 深度、buffering 方式，甚至 warp specialization 的结构。

这让 TileLang 在复杂内核上更有可能逼近人工专家内核的上限，但同时也意味着程序员必须重新承担更多复杂性。

### 8.7 为什么 TileLang 强，但也让人痛苦

TileLang 之所以强，是因为它重新打开了 Triton 曾经帮我们折叠掉的一部分优化空间。正因如此，它能够在复杂注意力、量化内核、跨硬件调度、特殊布局和高度融合的内核上走得更远。

但它之所以痛苦，也正因为此：内存布局、线程绑定、流水线、tensorization 等低层决策重新成为程序员必须直面的对象。

课程对此的判断其实很深刻：从“人机协同”的角度看，把这些低层旋钮重新交回给模型开发者，本身并不是理想状态。TileLang 是工程上对真实问题的有效回应，但从更深层看，它也意味着：自动化还不够强，所以人不得不重新接管更多细节。

## 9. CUDA-L2：让学习式搜索去击败人类启发式

### 9.1 CUDA-L2 在概念上为什么重要

CUDA-L2 不是“又一个 kernel optimizer”，而代表着一种不同方向：

> 与其把更多优化旋钮交还给人类，不如训练一个系统，让它在庞大的离散 kernel 设计空间里搜索得比人更有效。

CUDA-L2 关注的是 A100 上的半精度 GEMM。它把大语言模型和强化学习结合起来，在 1000 组 \((M,N,K)\) 配置上优化 HGEMM kernel，并在自己的评测设置中报告平均性能超过了 cuBLAS 和 cuBLASLt 的强基线。

这件事的重要性在于两点：

1. GEMM 是 LLM 中最典型的工作负载；
2. cuBLAS / cuBLASLt 已经凝结了多年专家工程经验。

如果一个学习式系统能在这种场景下取得优势，就说明：优化空间里仍然存在大量高质量解，但人类工程师要么没有系统地搜索到，要么因为启发式偏见过早把它们剪掉了。

### 9.2 CUDA-L2 由哪些部件组成

根据论文与课程总结，CUDA-L2 相比更早的 CUDA-L1，大致加入了这些关键成分：

1. 在更丰富 CUDA 代码语料上的持续预训练；
2. 多阶段 RL 训练，从更宽泛的 kernel 优化逐渐走向 matmul 专门化；
3. 把更丰富的 Nsight Compute profiling 指标送入上下文；
4. 用检索增强方式补充基础模型中没有充分编码的架构知识与优化经验。

这里真正起作用的 reward，是在真实硬件上编译、运行、测量之后得到的执行速度。因此，这不是纸面推理，而是一个被真实执行反馈闭环约束住的搜索系统。

### 9.3 为什么学习式搜索有机会超过专家

课程里对这一点解释得非常到位：人类专家并不搜索整个空间，人类搜索的是“被经验压缩过的空间”。

例如，人会天然带着一些先验：

- tile size 最好整除维度；
- padding 总归有点浪费；
- prefetch 不必看太远；
- 某种 swizzle 通常是标准答案；
- double buffering 大概率值得，或者大概率不值得；
- 中间 staging tensor 往往是必要的。

这些规则并不愚蠢，它们本身就是经验的压缩；但问题在于，它们也会在搜索开始前就把大量可能性剪掉。学习式系统之所以能赢，不一定是因为它“更聪明”，而是因为它愿意系统地尝试那些人类会直觉性忽略的选项。

### 9.4 CUDA-L2 找到的几个代表性案例

课程里列举的几个案例非常生动地说明了这一点。

第一，人类常坚持 tile size 最好整除矩阵维度；但 CUDA-L2 发现，轻微 padding 带来的局部损失，有时远小于更优 tile 带来的整体收益。例如某些维度下，160 这种看起来“不自然”的 tile 只需要很少 padding，却显著优于 128 或 256 这类更传统的选择。

第二，人类常默认只在 \(K\) 维前取一块数据做 prefetch；但 CUDA-L2 发现，在 A100 上，更深的 lookahead（例如 \(K+4\)）有时能更好地隐藏异步拷贝延迟。

第三，很多人会保守地引入一个中间 staging tensor 来做寄存器到共享内存的数据重排；但 CUDA-L2 找到了一些情况，在对齐与布局已经足够理想时，完全可以直接做宽向量 store，省掉中间 staging。

第四，double buffering 虽然经典，但并非总是有利。CUDA-L2 不只是学会“使用双缓冲”，也学会“什么时候不要用它”，尤其是在寄存器压力会压垮 occupancy 的场景下。

第五，它还会重新安排 prefetch A 和 prefetch B 的时机，把它们与计算更细致地交错，从而改善指令级并行性与内存交通平滑度。

第六，它还会针对小、中、大尺寸 GEMM 选择不同 CTA swizzle 策略，以提升 L2 命中率。这里的重点不是某种具体 swizzle，而是：最优策略可能根本不是全局统一的。

这些例子说明的不是“模型有神秘直觉”，而是：当搜索真正覆盖到那些人类先验通常不会系统探索的区域时，就可能发现经验规则之外的高质量配置。

### 9.5 CUDA-L2 真正教给我们的东西

CUDA-L2 至少说明了三件事：

1. 即使在 GEMM 这种被研究得极其充分的领域，优化空间仍然远未被穷尽；
2. 人类内核工程的限制不仅在知识量，也在搜索策略本身；
3. 面对越来越复杂的低层优化空间，长期更合理的答案，不一定是继续把旋钮交给人，而更可能是让机器去搜索这些旋钮。

## 10. 更广泛的新方向：用 AI 优化 AI 基础设施

CUDA-L2 并不是孤立工作，而属于一个更大的趋势。

课程最后一部分正确地指出，近年的方向大致可以分成几个簇：

- 专门的后训练内核模型；
- 代理式迭代搜索系统；
- 硬件感知 / profiling 感知搜索；
- 演化式或种群式搜索；
- 面向异构硬件的扩展；
- 以及基准和评测基础设施。

它们的共同点是：AI 基础设施本身，正在逐渐成为 AI 自动化优化的对象。编译器不再只是一个由人写死规则的优化系统，而越来越像一个环境：在这个环境里，学习型系统会提出候选、测量结果、修正策略、积累经验。

从历史角度看，这个转变非常重要。TVM 已经开始用机器学习去引导 autotuning；而 CUDA-L2 和后续系统更进一步，把低层优化本身当作一个推理—搜索问题来处理，并把 LLM 纳入这个循环。

课程里的那句总结其实很传神：

> 用“魔法”去打败“魔法”。

如果说得更朴素一点，那就是：**用机器学习来优化机器学习的执行过程本身。**

## 11. 把四类系统放在一起看

现在，我们可以更清楚地比较 TVM、Triton、TileLang 和 CUDA-L2。

### TVM

TVM 是最典型的“编译器中心型”系统。它假设问题是广泛的、异构的、多层次的，因此需要丰富的 IR 栈、compute-schedule 分离、图级优化、张量级调度以及自动调优来覆盖大空间。

### Triton

Triton 是“程序员友好，但仍由编译器托底”的系统。它假设工作负载和硬件都足够收敛，因此可以用一个更轻、更专的 DSL 暴露 block 级抽象，并在保留高性能的同时大幅降低开发复杂度。

### TileLang

TileLang 是“控制权回归”型系统。它仍以 tile 为核心，但重新暴露更多关于 memory layout、pipeline、thread binding 和 tensorization 的控制，因为 Triton 那种收缩后的抽象已经不足以覆盖高度特化的 kernel。

### CUDA-L2

CUDA-L2 是“学习式搜索”型系统。它的判断是：这个优化空间对人来说已经过大、过碎、过依赖 shape 与硬件，因而更合理的做法，是让 LLM + RL 直接在真实执行反馈下搜索这个空间。

所以，这四者并不仅仅是四个工具，而是四种不同的自动化落点。它们分别回答了：优化究竟应该更多地留给人、留给编译器，还是留给学习系统。

## 12. 本章真正的收获

本章最深的一层教训，并不在某一个具体机制，而在两个总体判断。

第一，**一个计算从来不只有一种表示，它必须拥有多个表示。**

- 模型层保留语义与设计意图；
- 图层保留全局依赖结构；
- 张量层保留 loop、buffer 与 schedule 空间；
- 硬件层保留执行与资源约束；
- 运行时层保留动态协调与启动行为。

不同层暴露不同结构，也正因为如此，不同优化才能在不同层上发生。编译器栈存在的理由就在这里。

第二，**内核优化本质上是一个结构化离散搜索问题。**

这就是为什么手工 CUDA 优化如此困难；也是为什么需要层次化编译器栈、为什么 compute-schedule 分离有意义、为什么 Triton 曾一度把问题大幅简化、为什么 TileLang 又不得不重新暴露控制、以及为什么 LLM + RL 这样的系统现在会变得如此有吸引力。

还有第三个更历史性的结论：深度学习编译器之所以不断变化，并不是因为编译器研究者“偏好变了”，而是因为世界本身变了。

- 当世界是异构的，我们需要通用编译器；
- 当世界收敛了，我们需要更高生产力的专用 DSL；
- 当专用化带来新的复杂度，我们又需要更细粒度的控制；
- 当这些控制已经细到人类难以系统管理时，我们又重新把希望寄托回自动化与学习式搜索。

这个循环还没有结束。更可能的未来，是把编译器的分层结构与学习系统的搜索能力合在一起：保留编译器擅长的结构化表达，再让学习型系统去接管那些人类难以稳定搜索的部分。

## 13. 最新进展

自 CUDA-L2 在 2025 年底出现之后，这个方向扩张得非常快，已经不再是一条单线前进的脉络，而是形成了几个相对清晰的分支。

第一类，是面向 kernel 生成本身的专门后训练模型。CUDA-L2 前后有 AutoTriton、CUDA-L1 等工作；而 2026 年之后，更多系统开始引入多轮 RL、profiling 奖励、更稳定的 credit assignment 机制，以及更强的合成数据管线。整体趋势是：kernel 生成不再被当成普通代码生成，而被视为一个独立的后训练任务域。

第二类，是不依赖重训练、而更强调闭环搜索的代理式系统。它们通常采用“生成候选代码 -> 编译 -> 正确性检查 -> profile -> 修改 -> 保留优者”的循环，把 kernel 优化明确建模成长期搜索过程，而不是一次性采样。

第三类，是硬件感知与 profiling 感知搜索。与其让模型盲目尝试，不如让它直接围绕 bottleneck、硬件结构和性能计数器来决策。例如围绕 swizzle、cache 命中率、speed-of-light 分析来指导搜索，这一方向正在快速增强。

第四类，是演化式或种群式搜索。其核心判断是：kernel 优化景观高度不规则，因此用一条单独的局部改进链往往不够，不如维护一批多样化候选，通过选择、变异、保留精英来推进。近年来这一类工作增长很快。

第五类，是异构硬件和非 CUDA 扩展。一个重要问题是，这个领域会不会长期被 NVIDIA / CUDA 锁死。目前答案越来越像“不会”：越来越多工作开始覆盖 AMD、CPU、Meta 自研加速器、不同 DSL 和不同近裸机 ISA。

与此同时，benchmark 与 evaluation 也越来越重要。KernelBench 逐渐成为核心公共基准，但大家也越来越清楚：比较对象是谁、搜索预算给多少、正确性验证是否稳健、是否与真正强基线比较，这些都会大幅改变“效果很好”这句话的含义。

总体来看，CUDA-L2 证明了“LLM + RL 能在狭窄而重要的 kernel 任务上取得很强结果”；而后续一波工作，则把这个思想扩展成更完整的代理式、profile 驱动、演化式和异构硬件导向的系统图景。

## 14. 给学生的一套紧凑心智模型

以后你再读任何一个深度学习编译器，都可以先问自己五个问题：

### 1. 它面向的到底是哪种问题区间？

它是在解决“多模型 × 多后端”的通用性问题，还是在解决“单一高价值工作负载在主流平台上的极致优化”？

### 2. 它主要工作在哪一层表示上？

图 IR、张量 IR、schedule IR、硬件 IR、运行时，还是几层的混合？

### 3. 在这一层里，哪些结构是可见的？

是全局依赖结构？是 loop nest？是 memory scope？是线程层次？还是 pipeline 形状？

### 4. 它把哪些优化旋钮暴露给了谁？

是暴露给程序员、暴露给 schedule 语言、暴露给代价模型，还是直接交给学习型智能体？

### 5. 它采用的搜索策略是什么？

是人工启发式？模板？代价模型？演化式搜索？强化学习？还是 LLM 代理式闭环？

如果这五个问题你都能答出来，那么这个系统的本质通常就已经很清楚了。

## 16. 全章总结

深度学习编译器之所以存在，是因为高层张量计算必须被映射到底层硬件执行上，而这种映射本身就是一个大规模、结构化的优化问题。优化旋钮——平铺、内存放置、循环顺序、融合、展开、线程映射、同步——共同构成了一个离散而耦合的搜索空间；这个空间大到无法在大规模场景下完全依靠人工完成。编译器栈存在的原因，就在于不同层保留了不同结构，而不同优化必须依赖不同层的可见性。

TVM 代表的是 LLM 之前那个“广谱、异构、多层”的答案：用丰富的 IR 栈、图级优化、张量级调度和 autotuning 去应对“很多模型 × 很多后端”的复杂世界。Triton 代表的是 LLM 时代早期的答案：在更收敛的工作负载与硬件条件下，用一个高生产力、高性能的 block 级 DSL 降低 GPU kernel 开发门槛。TileLang 则代表控制权重新回到用户手里：当 Triton 的黑盒抽象不够时，必须重新暴露 layout、pipeline、thread binding 与 tensorization。CUDA-L2 则代表再下一步：既然这些旋钮已经多到人类难以系统搜索，那就让学习系统在真实执行反馈下去搜索它们。

把这些系统放在一起看，我们看到的不是几种彼此无关的工具，而是一条非常清晰的历史演化线：问题的形状变了，于是编译器的形状也变了。未来更可能出现的，不是彻底抛弃编译器，也不是彻底回到人工专家，而是一种新的综合：保留编译器的层次化结构表达，再用学习系统去自动搜索其中最困难、最脆弱、最依赖硬件细节的那部分优化空间。
