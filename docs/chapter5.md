# 第 5 章 CUDA 编程作为软硬件协同优化

## 从朴素矩阵乘法到分层分块

本章将课程讲解与幻灯片内容重新组织为一条更清晰的叙述线索。全章围绕一个核心判断展开：**CUDA 编程本质上是映射问题**。高维张量工作负载必须被映射到线性、分层、资源受限的硬件结构上；而性能的高低，取决于这种映射是否尊重 GPU 的执行方式、存储层次与数据流规律。

延续上一章对 GPU 微架构的分析，我们已经知道：GPU 的性能并不只来自“算得快”，而是来自计算、存储、调度与通信的协同配合。`__global__`、线程块、网格、索引公式这些表层语法，只是编程入口；真正决定性能的，是高维数学结构如何落到 HBM、共享内存、寄存器以及 warp 执行单元上。

为把这个问题讲透，本章选取矩阵乘法作为贯穿全章的示例。原因很简单：矩阵乘法在数学上足够基础，在机器学习中又足够重要，同时它几乎会暴露 CUDA 优化里最关键的所有问题，包括访问模式、数据重用、共享内存平铺、寄存器平铺、向量化、warp 级组织以及共享内存存储体冲突。换句话说，矩阵乘法不是一个偶然的例子，而是理解 CUDA 编程的理想“样本”。

---

## 1. CUDA 编程首先是映射问题

从工作负载的角度看，机器学习中的核心计算几乎都建立在张量之上。矩阵是二维张量，批量矩阵、卷积张量、注意力张量则往往具有更高维的结构。这些数据在逻辑上天然带有多维邻接关系：矩阵元素有行邻居和列邻居，图像像素有空间邻居，批量张量则往往同时包含 batch、channel、高度与宽度等多个维度。

但硬件并不会直接“看到”这种逻辑结构。内存是线性编址的；DRAM 按行、列、页和突发传输组织；共享内存是容量有限、由软件显式管理的片上 SRAM；寄存器是线程私有的；而执行层面上，GPU 也不是按“二维几何对象”来运行程序，而是按一维的 warp 执行 32 个线程。因此，CUDA 编程一开始就面对一个根本张力：**工作负载是高维的，但硬件资源的组织方式是线性且分层的。**

围绕这一点，课程明确指出了三类核心错配。

第一类是**数据布局错配**。多维张量最终必须落到线性内存中，而一旦选定了某种布局，例如行主序，就意味着某些维度上的局部性会被保留，另一些维度上的局部性则会被削弱。

第二类是**执行错配**。程序员可以写二维甚至三维线程坐标，但真正执行时，硬件是按 warp 这个一维单位把线程编组并同步发射的。因此，`threadIdx.{x,y,z}` 到问题空间的映射绝不是语法细节，而是决定哪些线程会在同一个 warp 中相邻、哪些地址会被一起访问的关键。

第三类是**通信布局错配**。相同的算法，如果数据在错误的层级上被重复搬运，成本会截然不同。寄存器中的重用最便宜，共享内存中的重用次之，而反复回到 HBM 取数则成本高得多。算法本身并没有改变，但通信路径一旦不合理，观察到的性能就会急剧下降。

因此，本章后续所有优化都可以看成是在逐步消除这三类错配。只要抓住这一点，CUDA 优化就不再像一组零散技巧，而会呈现出一致的系统逻辑。

## 2. 运行示例：一个最朴素的矩阵乘法内核

为了让讨论具体落地，课程采用最经典的 GEMM 形式：

$$ C = A \times B + C, $$

其中 $A$、$B$、$C$ 都是按行主序存储的 $N \times N$ 矩阵。最自然的做法是：为输出矩阵中的每个元素 $C_{ij}$ 分配一个线程。该线程负责读取 $A$ 的第 $i$ 行、$B$ 的第 $j$ 列，完成点积，再把结果写回 $C_{ij}$。

最朴素的 CUDA 写法如下：

```text
__global__ void sgemm_naive(
    int M, int N, int K,
    float alpha,
    const float* A,
    const float* B,
    float beta,
    float* C) {

    const unsigned int x = blockIdx.x * blockDim.x + threadIdx.x;
    const unsigned int y = blockIdx.y * blockDim.y + threadIdx.y;

    if (x < M && y < N) {
        float tmp = 0.0f;
        for (int k = 0; k < K; ++k) {
            tmp += A[x * K + k] * B[k * N + y];
        }
        C[x * N + y] = alpha * tmp + beta * C[x * N + y];
    }
}
```

这段代码在功能上没有问题，逻辑上也非常直观：输出是二维的，于是使用二维网格和二维线程块；每个线程负责一个输出元素；内层循环就是标准点积。若从算法课的角度看，这已经是一份“完成度很高”的实现。

但在 GPU 编程里，这只能算起点。因为**正确性并不意味着高性能**。真正的问题不在于算式有没有写对，而在于这种线程到数据的映射方式，是否让机器看到了高效的数据流。

## 3. 理论上，矩阵乘法应当是计算受限的

课程接着提出一个极具启发性的问题：矩阵乘法究竟应该是计算受限，还是带宽受限？

从理论上说，答案非常明确。矩阵乘法的算术工作量随规模按 $O(N^3)$ 增长，而数据规模仅按 $O(N^2)$ 增长。也就是说，随着问题规模增大，其算术强度会不断上升。这正是 GPU 擅长的工作负载形态。

课程以 H100 SXM 和 $4096 \times 4096$ 的 FP32 矩阵乘法为例。H100 大约提供 80 GB HBM3，峰值带宽约为 $4.096\ \text{TB/s}$，FP32 CUDA Core 峰值吞吐量约为 $60\ \text{TFLOP/s}$。

对于

$$ A, B, C \in \mathbb{R}^{4096 \times 4096}, $$

总浮点计算量约为

$$ 2 \cdot 4096^3 + 4096^2 \approx 137\ \text{GFLOPs}. $$

这里的系数 2 对应点积中的乘法与加法，额外的 $4096^2$ 则对应将结果与原有 $C$ 相加。

从数据移动角度看，至少需要读取 $A$、$B$、原始 $C$ 三个矩阵，共约

$$ 3 \cdot 4096^2 \cdot 4\ \text{B} \approx 201\ \text{MB}, $$

再加上写回输出矩阵

$$ 4096^2 \cdot 4\ \text{B} \approx 67\ \text{MB}. $$

因此理论上的最小数据流量约为 $268\ \text{MB}$。

据此可以得到两个粗略下界：

- 如果完全受 HBM 带宽限制，时间下界约为 $268\ \text{MB} / 4.096\ \text{TB/s} \approx 0.07\ \text{ms}$。
- 如果完全受 FP32 计算吞吐限制，时间下界约为 $137\ \text{GFLOPs} / 60\ \text{TFLOP/s} \approx 2.3\ \text{ms}$。

也就是说，从 roofline 的角度看，GEMM 理论上应该更靠近计算屋顶，而不是内存屋顶：

$$ P \le \min(P_{\text{peak}}, BW_{\text{peak}} \cdot I), $$

其中 $P_{\text{peak}}$ 为峰值算力，$BW_{\text{peak}}$ 为峰值带宽，$I$ 为算术强度。对于 GEMM，$I$ 足够高，因此它本应是一个典型的计算密集型内核。

问题就在这里：理论如此漂亮，但朴素内核并没有跑出应有的表现。

## 4. 理论与现实之间：两个数量级的落差

课程随即给出实际测量结果：在 H100 上，这个朴素内核只有大约 $500\ \text{GFLOP/s}$ 的吞吐量，运行一个 $4096 \times 4096$ 的示例要超过 200 毫秒。

这个结果不是“稍微慢一点”，而是比理论峰值低了两个数量级以上。于是课程把问题重新表述为：算法正确、并行性充足、理论算术强度也很高，为什么现实性能却如此糟糕？

关键原因在于：纸面分析计算的是**逻辑数据流量**，而不是实现过程中真正发生的**物理数据流量**。理论估计默认每次内存传输都带来了有用数据，也默认需要重用的数据确实被重用了。但朴素内核同时违背了这两个假设：它既产生了糟糕的访问模式，浪费了带宽，也没有把本可共享的数据真正共享起来。

一旦实现层面的访问模式有问题，原本理论上“计算受限”的工作负载，也会在现实中退化为一个低效的“内存受限”内核。课程正是借这个例子提醒学生：**roofline 预测的是上限，而不是自动实现的结果。**

因此，下一步不能简单地说“用共享内存试试”，而必须先回到更底层的问题：GPU 的内存系统究竟喜欢什么样的访问方式？

## 5. DRAM、HBM 与合并访问的物理基础

课程这里花了较多篇幅回到 DRAM 本身，这是非常必要的。因为如果不了解 DRAM 的物理组织方式，“合并访问”就很容易被误解成一条需要死记的 CUDA 口诀。

在电路层面，DRAM 单元本质上是“电容 + 晶体管”。读取数据时，需要把一个很小的存储电荷耦合到位线中，再由感放器放大；而且这种读取通常还是破坏性的，因此读完之后还必须恢复原值。正因为单个单元的读取既微弱又昂贵，DRAM 从来不是为“随机取一个标量字”而优化的，而是为“按行激活，再做突发传输”而优化的。

也就是说，真正昂贵的操作不是“取一个 float”，而是“打开一整行”。一旦某一行被激活，后续在这一行附近进行连续访问就会便宜得多；反之，如果请求不断跨行跳转，系统就要频繁预充电、激活新行、重新感测和恢复数据，效率会显著下降。

HBM 继承了这一基本逻辑，只是通过更多 bank、更高并行度、更短布线距离和更高集成度，把总带宽推到了非常高的量级。但无论带宽多高，想要真正吃满它，软件都必须给出适合突发传输与行缓冲区的访问模式。

这就是“步幅”为什么如此重要。课程中用一个实验说明：小步幅访问的带宽可达到约 $1418\ \text{GB/s}$，而病态的大步幅访问则会掉到约 $111\ \text{GB/s}$，只有前者的不到十分之一。这个差距并不神秘，它本质上就是“每次打开一行之后，你到底利用了多少有效数据”的差距。

所谓**合并访问（coalesced access）**，说到底就是让一个 warp 发出的请求在地址空间中尽量连续，使硬件可以用更少、更密集、更有效的事务去服务这些访问。若 warp 中 32 个线程请求的是连续地址，那么一次突发传输就可能携带大量有用数据；若 32 个线程请求分散、跨步甚至彼此相隔很远，那么系统就会搬运大量几乎没被利用的数据。

所以课程在这里给出的第一个真正诊断视角是：不要只问“一个线程访问了什么”，而要问“**一个 warp 一起访问了什么**”。

## 6. 为什么线程局部性不够：还必须看 warp 局部性

这正是朴素 GEMM 内核失效的关键。程序员经常用二维线程块来对应二维输出矩阵，但硬件真正执行的是 warp，而 warp 中线程的编组顺序首先沿 `threadIdx.x`，再沿 `threadIdx.y`，最后才是 `threadIdx.z`。

因此，内存系统看到的不是某个线程独自的访问轨迹，而是一个 warp 中 32 个线程在同一时刻的联合访问模式。某个单线程看起来“很合理”的访问，在 warp 级别上完全可能是灾难性的。

回到朴素矩阵乘法。如果只看单个线程计算 $C_{ij}$，它在内层循环中沿着 $A$ 的第 $i$ 行前进，这对行主序来说是连续访问；而对 $B$ 来说，它沿着第 $j$ 列前进，是明显的跨步访问。从单线程视角出发，人们很容易得出“$A$ 访问不错，$B$ 访问糟糕”的判断。

但这不是 GPU 真正在意的视角。

如果一个 warp 中相邻线程对应的是输出矩阵中**相邻的行**，而列索引 $j$ 固定，那么在内层某个固定的 $k$ 上，这 32 个线程访问的是：

- $A[i, k], A[i+1, k], A[i+2, k], \dots$，也就是 $A$ 的同一列；
- 同时访问相同的 $B[k, j]$。

这意味着，从 warp 的角度看，对 $A$ 的访问实际上是大跨度的，对 HBM 极不友好；而对 $B$ 的访问则更像广播。这就解释了一个非常容易误判的问题：**线程局部性并不足以判断全局内存效率，真正关键的是 warp 局部性。**

课程在这里给出的教学价值非常高，因为它迫使学生从“每个线程做什么”转向“warp 这个硬件执行单位在请求什么”。这是 GPU 编程思维的第一次真正转弯。

## 7. 第一个关键优化：让 warp 沿行而不是沿列展开

既然问题出在 warp 的访问模式上，修复思路其实非常直接：不要让一个 warp 覆盖输出矩阵的一列条带，而应让它覆盖一行条带。

这样一来，在固定的 $k$ 上，如果 warp 中所有线程对应同一输出行 $i$，只是列索引变为相邻的 $j, j+1, \dots, j+31$，那么：

- 所有线程都需要同一个标量 $A[i,k]$，这天然具有广播特征；
- 同时它们需要的是连续的 $B[k,j], B[k,j+1], \dots, B[k,j+31]$，也就是 $B$ 某一行上的连续片段。

这正是 HBM 最喜欢的模式：一个操作数更接近广播，另一个操作数则可以合并访问。课程里对应的实验结果也说明了这一点：仅仅通过调整线程到输出元素的映射方式，性能就从约 $0.5\ \text{TFLOP/s}$ 提升到约 $6.3\ \text{TFLOP/s}$，差不多就是一个数量级以上的提升。

这里尤其值得强调的是：算法没有变，内层循环没有变，数学工作量也没有变，变化的只是索引映射。正因为这类修改“代码很小、效果很大”，很多初学者才会误以为 CUDA 优化像是在玩魔法。实际上并不是魔法，而是索引映射背后编码了大量关于 warp 组织与 HBM 事务的物理事实。

从 roofline 角度看，这一步没有改变 GEMM 的理论算术强度，却显著改善了**实现层面的算术强度**。因为每个内存事务携带的有用数据更多了，带宽浪费更少了，内核自然就开始向计算屋顶移动。

## 8. 下一层瓶颈：合并访问解决了 HBM，却没有解决重用

这一轮优化虽然惊人，但离 H100 的峰值仍然很远。于是课程进一步指出：修复全局内存合并访问之后，下一个瓶颈变成了**数据没有被共享**。

问题在于，当前内核虽然让每个线程更高效地从 HBM 中读取了数据，但这些数据仍然被直接读入线程私有寄存器中。寄存器不能在线程之间共享，因此即使相邻线程需要 $A$ 或 $B$ 的重叠部分，这种重叠也没有被真正利用。不同线程会反复从 HBM 把相同的值各自加载一遍。

从算法层面看，GEMM 应当是“搬运 $O(N^2)$ 数据、执行 $O(N^3)$ 运算”；但若每个线程都私有地加载重叠操作数，实现层面的全局内存流量就会明显膨胀。换句话说，**算法层面的数据重用机会没有在实现层面兑现出来。**

这正是共享内存登场的理由。

## 9. 共享内存平铺：把块内重用真正做出来

共享内存是一个容量较小、速度较快、块内线程可见的片上存储。与寄存器不同，它允许线程块内部进行显式协作，因此天然适合承载块级别的数据重用。

所谓**共享内存平铺（tiling）**，其思想非常朴素：既然某个线程块负责输出矩阵 $C$ 的一个子块，那么该线程块中的线程必然会共同使用来自 $A$ 和 $B$ 的某些子块。与其让每个线程都独立去 HBM 里取各自需要的数据，不如先让线程块协作，把这些输入子块搬到共享内存中，再由块内各线程反复重用。

课程中的过程可以概括为：

1. 为输出矩阵划分线程块负责的输出 tile；
2. 确定计算该输出 tile 所需的 $A$、$B$ 输入 tile；
3. 由块内线程协作，把这些输入 tile 从全局内存搬入共享内存；
4. 在共享内存中完成若干轮局部乘加累积；
5. 沿着 $K$ 维重复上述过程；
6. 最终把累积结果写回全局内存中的输出 tile。

示意性代码如下：

```text
__shared__ float As[BM][BK];
__shared__ float Bs[BK][BN];

float acc = 0.0f;

for (int tile = 0; tile < K; tile += BK) {
    As[ty][tx] = A[row * K + (tile + tx)];
    Bs[ty][tx] = B[(tile + ty) * N + col];
    __syncthreads();

    for (int k = 0; k < BK; ++k) {
        acc += As[ty][k] * Bs[k][tx];
    }
    __syncthreads();
}
```

这里的关键不在具体索引，而在通信结构的变化：来自 HBM 的每一个输入值，不再只服务一个线程，而是会在块内被多个线程反复利用。于是同样的 HBM 流量可以支撑更多算术操作，全局内存层面的算术强度随之上升。

课程中的测量结果表明，加入共享内存平铺后，性能从约 $6.3\ \text{TFLOP/s}$ 进一步提升到约 $9\ \text{TFLOP/s}$。这是一个非常重要的里程碑，因为它说明：一旦把数据重用从“算法上的可能性”变成“实现中的现实”，性能就会出现明显跃升。

更深一层的系统意义在于：共享内存不是自动缓存，而是由软件显式管理的存储层。这意味着程序员或编译器必须主动决定“什么数据值得放进来、什么时候搬进来、在块内如何组织”。从这一点开始，CUDA 编程就已经不只是并行语法，而是显式的数据编排。

## 10. 共享内存更快，但它也会成为新的瓶颈

优化不会消灭瓶颈，它只会把瓶颈推向下一个层级。共享内存平铺之后，课程观察到，内核虽然不再主要受 HBM 限制，但新的问题出现了：**共享内存本身的访问压力过大。**

原因并不复杂。在最基础的共享内存平铺实现中，每个线程仍然只计算一个输出元素。于是每进行一次乘加，该线程都需要从共享内存里取一项来自 $A$ 的值和一项来自 $B$ 的值，再送入寄存器做 FMA。共享内存虽然比 HBM 快得多，但如果每个 FLOP 都伴随大量共享内存读取，那么共享内存指令流量本身就会成为瓶颈。

课程中的停顿分析也指出，这类内核常见的 stall 原因是 MIO（memory input/output）队列过满。也就是说，问题已经不再是“全局内存是否连续”，而是“共享内存指令是否过于密集”。

到这一步，优化目标也随之变化：不再是进一步减少 HBM 流量，而是要减少**每单位算术操作对应的共享内存流量**。这正是寄存器平铺的出发点。

---

## 11. 寄存器平铺：让每个线程计算更多输出

寄存器是 GPU 层次结构中延迟最低、带宽最高的存储层。课程在这里提出的核心思想是：如果想减少共享内存访问，就必须让每个线程在寄存器里做更多工作。

### 11.1 一维寄存器平铺

在最简单的一维寄存器平铺中，每个线程不再只负责一个输出元素，而是负责一小段输出。例如，它可以负责同一行上的多个相邻输出，或者同一列上的多个相邻输出。这样一来，从共享内存中加载来的一个操作数片段，就可以在多个寄存器累加器之间重复利用。

示意上，线程会维护一个寄存器数组：

```text
float threadResults[TM] = {0.0f};
```

随后，线程在一次共享内存读取之后，不再只做一个 FMA，而是对这个寄存器数组中的多个累加器做更新。这样，每个共享内存读出的值就被“榨出了更多算术”。

课程中给出的结果显示，一维寄存器平铺把性能从约 $9\ \text{TFLOP/s}$ 推高到接近 $20\ \text{TFLOP/s}$。这说明共享内存带宽瓶颈得到显著缓解。

### 11.2 二维寄存器平铺

进一步地，每个线程还可以在寄存器里维护一个小的二维输出块，而不是一维向量。此时，线程不再只是“计算多个元素”，而是更像在执行一个小型矩阵乘法微内核：加载一小片来自 $A$ 的片段与一小片来自 $B$ 的片段，在寄存器中完成一个小外积更新。

二维寄存器平铺进一步提高了寄存器级重用，使 $A$ 端与 $B$ 端的共享内存数据都可以在线程内部得到更多利用。课程中给出的性能也相应提升到约 $25.8\ \text{TFLOP/s}$。

### 11.3 为什么寄存器平铺有效

寄存器平铺之所以重要，是因为它把“数据重用”又向层次结构内部推进了一层。

- 合并访问解决的是 HBM 事务效率；
- 共享内存平铺解决的是块内重用；
- 寄存器平铺解决的是线程内重用。

也就是说，数据正在逐步被放到更便宜的层级上反复使用。这正是高性能 GPU 内核的基本逻辑。

当然，这里也带来了新的权衡：寄存器平铺意味着每个线程持有更多寄存器，从而可能降低占用率，因为一个 SM 上能同时驻留的线程块或 warp 数量会减少。因此，最优 tile 大小从来不是固定不变的“神奇数字”，而是共享内存使用、寄存器压力、占用率与调度灵活性的折中结果。

从这里开始，CUDA 编程已经非常像系统设计，而不再只是“把循环并行化”。

---

## 12. 向量化访问：让每条指令搬更多数据

在解决了 HBM 局部性、块级重用和线程级重用之后，课程接着讨论了另一个看似细节、实则很重要的优化：**向量化内存访问**。

即使访问已经是合并的，如果每次仍只加载一个标量，那么内存指令的数量仍然可能偏多。只要对齐条件允许，就可以使用 `float2`、`float4` 之类的向量化加载，以减少指令条数并提高每条指令承载的数据量。

课程还强调了共享内存的另一个重要作用：它不仅是更快的“中转站”，也可以是**布局转换缓冲区**。例如，一个操作数在全局内存里天然按行组织，而另一个操作数在后续计算中则更适合按列或按别的方式访问。此时，软件可以在把数据从全局内存搬入共享内存的过程中，顺便完成布局重排，使得后续从共享内存进入寄存器的访问更局部、更规则、更容易向量化。

这说明共享内存的价值不仅在“缓存”上，更在“重塑数据形状”上。它让数据不仅被搬进来，而且被搬成对下一级更友好的格式。

---

## 13. warp 平铺：把 warp 本身视为计算对象

到这里，课程再向前推进一步，提出了 **warp 平铺** 的概念。

此前的讨论虽然已经频繁提到 warp，但主要仍以“块”和“线程”为视角组织工作。然而硬件真正调度和执行的单位是 warp，因此成熟的实现不应只把任务分配给线程块和线程，也应把任务分配给**warp 这个集体执行单元**。

所谓 warp 平铺，就是让一个 warp 协同计算输出矩阵中的一个子块，而不是让每个线程仅仅独立计算一个标量输出。这样做的好处在于：

- 计算组织方式更贴近真实执行单元；
- 某些同步开销可以减少；
- 寄存器与共享内存中的重用可以进一步围绕 warp 来组织；
- 算术强度和局部性都有机会继续改善。

如果从全章逻辑回看，这一步非常自然。前面讲“执行错配”的时候，我们已经知道二维线程抽象不等于真实执行结构；到了这里，课程其实是在说：既然硬件按 warp 执行，那么软件也应越来越显式地围绕 warp 来组织计算。

在成熟的 GEMM 内核中，常见的层次结构就是：线程块负责一个较大的 tile，warp 在其中负责更小的子 tile，而单个线程再在寄存器中持有更细粒度的结果块。这个三层结构与 GPU 的真实存储和执行层次是高度一致的。

---

## 14. 共享内存存储体冲突：为什么“快内存”也可能变慢

当共享内存被大量使用之后，课程引入了另一个常见细节：**存储体冲突（bank conflict）**。

共享内存之所以快，是因为它是片上 SRAM；但它并不是一个“无限并行、毫无结构”的黑盒。以 H100 为例，共享内存可以看作由 32 个存储体组成。如果一个 warp 的 32 个线程分别访问不同的 bank，那么这一轮访问可以高效完成；若多个线程落到同一个 bank 上，请求就会串行化，访问延迟相应上升。

因此，共享内存的性能不只取决于“有没有放进共享内存”，还取决于**warp 如何映射到共享内存的物理存储体上**。这也是为什么有时程序员会在共享内存数组中做 padding：例如，当每行正好宽度为 32，而 warp 又按列读取时，就很容易让所有线程撞到同一个 bank 上；如果每行额外补一个元素，把步幅从 32 改成 33，病态对齐关系就被打破，冲突也会显著减少。

共享内存到这一步为止，已经不再只是“高速缓存替代品”这么简单，而是变成了一个既要考虑容量、又要考虑布局、还要考虑物理映射的精细化资源。

---

## 15. 把所有优化看成一条统一的阶梯

课程在这一阶段已经介绍了一长串技术：合并访问、共享内存平铺、寄存器平铺、向量化、warp 平铺、存储体冲突规避。若把它们当作互不相关的技巧去记忆，很容易越学越碎。

更好的理解方式，是把它们看成一条逐级推进的优化阶梯：

- **全局内存合并访问**修复了数据布局与 warp 访问顺序之间的错配；
- **共享内存平铺**修复了块级数据重用未被兑现的问题；
- **寄存器平铺**修复了共享内存带宽压力过大的问题；
- **向量化**修复了标量访存粒度与内存系统偏好传输宽度之间的错配；
- **warp 平铺**修复了逻辑线程组织与真实执行单元之间的错配；
- **存储体冲突管理**修复了逻辑共享内存布局与片上 SRAM 物理结构之间的错配。

从这个视角看，本章的每一项优化做的都是同一件事：**把工作负载的自然结构重新整理为更符合机器组织方式的结构。**

因此，CUDA 优化绝不是“堆技巧”，而是不断消除映射错配的过程。

---

## 16. 为什么仍然很难击败 cuBLAS

课程最后指出，即便经过这一系列优化，基于 CUDA Core 的 GEMM 也只是达到了约 $25$ 到 $26\ \text{TFLOP/s}$。这相较于最初的 $0.5\ \text{TFLOP/s}$ 已经是近两个数量级的提升，但距离 H100 的理论上限仍有明显差距，更不必说使用 Tensor Core 的高性能路径。

课程中半开玩笑地说：在它自己的主场上，想击败 cuBLAS 并不容易。这句话背后其实有非常扎实的系统原因。

首先，像 cuBLAS 这样的生产级库，不只是采用了我们课堂上讨论的这些思想，而是把它们做到了极致：针对不同架构、不同问题规模、不同数据类型，精细选择 tile 大小、线程块布局、寄存器使用与调度策略。

其次，它们大量使用软件流水线和双缓冲，让加载下一块数据与当前块上的计算更充分地重叠。在新架构上，像 `cp.async` 这样的异步拷贝机制会把这种重叠进一步做深。

第三，生产库往往优先走 Tensor Core 路径，而不是单纯依赖 CUDA Core。后者虽然更容易教学，但前者在数据格式与 tile 匹配时可以提供远高于 CUDA Core 的吞吐。

第四，库级实现还会结合更深层的架构特定优化，例如特殊的指令选择、寄存器 blocking、布局转换和调度细节。这些优化很难靠手工在短时间内完整复现。

因此，本章真正要传达的，不是“我们的内核还不够强”，而是“高性能库为什么值得被尊重”。当学生亲眼看到一个 GEMM 内核需要同时协调多少层资源，才会真正理解为什么自动调优、编译器搜索与成熟库会在机器学习系统中扮演如此重要的角色。

---

## 17. 本章真正想教会我们的是什么

如果要用更抽象的方式概括本章，可以把它压缩成以下几条核心教训。

第一，**理论算术强度不等于实现算术强度**。GEMM 在数学上是计算密集型，但糟糕的访问模式和重用方式会让它在现实中表现得像一个低效的内存内核。

第二，**优化的基本单位不是单个线程，而是整个层次结构**。真正需要考虑的单位包括 HBM 事务、warp、线程块、共享内存 tile、寄存器 tile 以及执行管线。

第三，**数据重用必须不断向更内层推进**。在 HBM 中重用最昂贵，在共享内存中重用较好，在寄存器中重用最好。高性能内核就是不断把重用推向更便宜层级的过程。

第四，**工作负载的几何结构并不等于执行结构**。矩阵是二维的，但 warp 是一维的。任何忽略这一点的映射方式，都可能从一开始就浪费大量带宽。

第五，**很小的代码改动可能对应很大的硬件事实**。一次看似不起眼的索引调整，背后可能是在重新组织整个 warp 的访存事务。

第六，**性能调优是逐层暴露瓶颈、逐层解决瓶颈的过程**。先是 HBM 访问效率，接着是块级重用，再接着是共享内存带宽、寄存器压力、warp 组织和存储体映射。GPU 优化几乎从来没有“一招制胜”。

这也正是 CUDA 编程为什么应该出现在机器学习系统课程中。它告诉我们：性能从来不只是算法的属性，而是算法在特定硬件约束下被实现出来之后的系统属性。

---

## 18. 结论：CUDA 编程就是让映射尊重机器

现在，我们终于可以回到本章最核心的判断：

**CUDA 编程就是把高维工作负载映射到线性、分层且资源受限的硬件之上。**

这不是一句口号，而是本章所有优化背后的统一原则。

当 warp 的访问模式尊重 HBM 的突发传输与行缓冲区时，就会出现合并访问；当块级重用被显式搬进共享内存时，就会出现共享内存平铺；当共享内存本身成为瓶颈时，就会出现寄存器平铺；当程序员真正把 warp 当作执行单元来组织计算时，就会出现 warp 平铺；当人们意识到共享内存内部也有物理结构时，就会认真处理存储体冲突。

因此，CUDA 编程的美感恰恰在于：每一个有效优化背后，都有明确的物理和架构解释；每一个糟糕实现背后，也都对应着具体可分析的系统成本。软件并不是在“写出算术”就结束了，而是在把算术塑造成机器真正高效的形式。

也正因为如此，本章自然通向机器学习编译器。一旦人们看清楚：布局、warp 映射、平铺、同步、向量化、存储体映射、寄存器压力、占用率这些因素会以如此复杂的方式相互耦合，就会明白自动内核生成与自动调优并不是锦上添花，而几乎是不可避免的下一步。

所以，真正正确的最终心智模型，不是“CUDA 让我启动很多线程”，而是：

**CUDA 让我设计数学结构与机器结构之间的映射。**

映射越尊重机器，性能就越接近理论所允许的上界。
