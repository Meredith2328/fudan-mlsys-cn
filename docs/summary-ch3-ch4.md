### 核心主线

这两章共同回答了一个关键问题：**为什么“会写 CUDA”并不等于“会做 GPU 优化”**。  
真正的答案是：GPU 性能来自对**工作负载结构、硬件执行模型、内存层次与编译器行为**的联合理解。

- **第3章** 从历史与架构演进出发，解释了 GPU 为什么会变成今天这种“吞吐量机器”。
- **第4章** 从编程与执行细节出发，解释了在这种机器上应当如何思考与优化 CUDA 内核。

---

### 第3章核心总结：GPU 架构与机器学习系统

这一章的核心不是“背硬件参数”，而是建立**因果链**：  
工作负载变化 -> 架构目标变化 -> 硬件组织变化 -> 软件抽象变化。

#### 1. GPU 胜出的根因是工作负载匹配

1. CPU 优先优化单线程延迟，GPU 优先优化整体吞吐量。  
2. 当工作负载是“大量相似、可并行的数值计算”时，GPU 的设计更合适。  
3. 机器学习（尤其是张量计算）天然满足这种结构，因此与 GPU 高度匹配。

#### 2. GPU 发展不是一次性发明，而是三代演进

1. **固定功能图形时代**：只能跑预定义图形管线。  
2. **可编程着色器时代**：开始能写程序，但编程模型仍受图形语义束缚。  
3. **通用并行计算时代**：CUDA 等抽象出现，GPU 真正成为通用高吞吐平台。

#### 3. 从 GPGPU hack 到 CUDA 的转折

1. 早期通用计算要“伪装成图形任务”，开发复杂且脆弱。  
2. Brook 与 CUDA 的价值在于提供了正确抽象：流、内核、线程层次。  
3. 本质是软硬件协同设计：硬件给并行能力，软件抽象把能力变成可用生产力。

#### 4. 现代 GPU 的关键架构思想

1. **SM + Warp + SIMT**：硬件以 warp 为真实调度单位。  
2. **存储层次驱动性能**：寄存器/共享内存/L2/全局内存的延迟差决定优化空间。  
3. **张量核心与数据移动优化并重**：不是只追 FLOPS，而是追“可持续供数 + 可持续计算”。

### 第4章核心总结：CUDA 编程从架构开始

这一章强调：CUDA 优化首先是**系统映射问题**，不是语法问题。

#### 1. CUDA 程序总是主机-设备协作系统

1. GPU 是加速器，不是孤立世界。  
2. 端到端时间包含：分配、拷贝、启动、回传，而不只是 kernel 时间。  
3. 算术强度低的任务即使“可并行”，也可能被数据传输成本压垮。

#### 2. Kernel 本质是“线程空间到数据空间”的映射

1. 线程索引公式决定每个线程负责哪块数据。  
2. Grid-Block-Thread 不是仪式，而是映射到硬件与内存层次的结构。  
3. 映射质量直接决定访存合并、局部性与最终吞吐。

#### 3. 性能核心在内存，不在算术口号

1. 高速存储小、慢速存储大，因此必须平铺（tiling）。  
2. 共享内存用于块内重用，寄存器用于线程私有热数据。  
3. 优化目标是减少无效数据移动，让算术单元持续吃满。

#### 4. Warp 语义决定控制流与并发效果

1. Warp 是执行基本单元，不是单线程。  
2. 分支发散会导致同一 warp 路径串行化，降低有效并行效率。  
3. 因此控制流、数据布局和任务切分都要“按 warp 友好”来设计。

#### 5. 编译器与硬件边界：PTX、SASS、寄存器与占用率

1. PTX 代表编译器中间意图，SASS 是最终机器执行现实。  
2. 寄存器过多会压低 occupancy，过少可能导致 spilling。  
3. 性能调优本质是平衡：寄存器压力、并发线程数、内存流量、指令效率。

### 两章的内在联系

- **第3章回答“为什么是这台机器”**：GPU 为什么长成现在这样。  
- **第4章回答“如何在这台机器上写对程序”**：如何把算法映射到真实执行模型。  
- 二者合在一起形成一个闭环：  
  **工作负载结构 -> 架构设计 -> 编程模型 -> 编译与执行行为 -> 实际性能结果**。

因此，3-4 章的共同结论是：  
**CUDA 优化不是“调几个参数”，而是在架构约束下做端到端的系统设计。**
